# 安全政策 / Security Policy

**最后更新：2026年4月**

---

## 安全漏洞报告

### 如何报告

如果您发现安全漏洞，请**不要**通过公开的 GitHub Issues 报告。请使用以下负责任披露方式：

1. **发送邮件**至项目安全团队的安全邮箱（参见仓库 About 页面）
2. **使用 GitHub Security Advisories**：在仓库的 Security 标签页提交私有安全报告

### 报告内容

请在报告中包含以下信息：

- 漏洞类型（如：数据泄露、注入攻击、权限绕过等）
- 受影响的项目版本
- 复现步骤
- 潜在影响
- 如有可能，提供修复建议

### 响应时间

| 阶段 | 时间 |
|------|------|
| 确认收到报告 | 48 小时内 |
| 初步评估 | 5 个工作日内 |
| 修复方案确认 | 10 个工作日内 |
| 补丁发布 | 30 天内（严重漏洞 7 天内） |

### 披露政策

- 在补丁发布前，报告者和项目团队均不得公开漏洞详情
- 补丁发布后，项目团队将在安全公告中披露漏洞信息
- 我们感谢负责任披露，将在安全公告中致谢（除非报告者要求匿名）

## 数据处理安全

### 数据分类

| 分类 | 说明 | 保护措施 |
|------|------|----------|
| 公开数据 | 已公开的政策文本 | 标准保护 |
| 内部数据 | 系统配置、来源列表 | 访问控制 |
| 敏感数据 | API 密钥、连接字符串 | 加密存储，日志脱敏 |
| 禁止数据 | 涉密信息、个人信息 | 不得存储，发现即清除 |

### 数据存储

- 所有数据文件存储在配置的数据目录中，不得硬编码路径
- 数据目录权限设置为仅项目用户可访问
- 临时文件在使用后立即清除
- 数据目录不纳入版本控制

### 数据传输

- 所有外部通信使用 HTTPS/TLS 加密
- API 调用不传输敏感数据（密钥通过环境变量注入，不随请求发送）
- 如需跨境数据传输（如调用境外 API），须评估合规风险

## API 密钥保护

### 核心原则

**API 密钥、令牌、凭证不得硬编码在源代码中，不得提交至版本控制系统。**

### 具体规则

1. **环境变量**：所有密钥通过环境变量或 `.env` 文件注入
2. **`.env` 文件**：已在 `.gitignore` 中排除，不得提交
3. **`.env.example`**：提供模板文件，仅包含变量名和占位符
4. **代码中**：使用 `python-dotenv` 加载，不直接引用密钥值
5. **日志中**：密钥在日志输出中自动脱敏

### 正确示例

```python
import os
from dotenv import load_dotenv

load_dotenv()

api_key = os.getenv("CPI_LLM_API_KEY")
```

### 错误示例

```python
# 禁止
api_key = "sk-xxxxxxxxxxxx"
# 禁止
api_key = "CPI_LLM_API_KEY=sk-xxxx"  # 不要在代码中包含实际值
```

### 泄露响应

如发现密钥泄露：
1. 立即轮换（rotate）泄露的密钥
2. 使用 `git filter-branch` 或 BFG 从 git 历史中清除
3. 在安全公告中披露事件（不包含密钥值）
4. 评估泄露影响范围并通知相关方

## 依赖安全

- 定期运行 `pip audit` 检查已知漏洞
- 使用 `pip freeze` 锁定依赖版本
- 更新依赖前评估安全影响
- 仅使用可信来源的依赖包

---

## 安全审计

- 每季度进行一次安全自审计
- 审计内容包括：
  - 代码中的硬编码凭证扫描
  - 依赖漏洞检查
  - 数据目录权限验证
  - 日志中的敏感信息扫描
- 审计结果在项目内部分享
