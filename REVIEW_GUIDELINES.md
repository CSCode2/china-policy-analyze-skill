# 评审指南 / Review Guidelines

**最后更新：2026年4月**

---

本文件规定了项目 Pull Request 的 8 项评审要点。所有 PR 必须通过以下检查清单后方可合并。

---

## 评审清单

### 1. 来源可追溯性检查

- [ ] 所有新增的分析逻辑是否有对应的数据来源标注
- [ ] 输出中是否包含来源 URL、发布机构、发布日期
- [ ] 证据等级标注是否准确（L1–L5）
- [ ] 不存在无来源的结论性陈述

### 2. 证据等级合规性

- [ ] 核心结论是否基于 L3 或以上证据等级
- [ ] L4（行业共识）和 L5（专家研判）是否标注了主观性声明
- [ ] 不存在将低等级证据作为高等级证据引用的情况
- [ ] 证据等级的判断依据是否在代码或配置中可查

### 3. 敏感内容合规

- [ ] 变更是否涉及军事/国防/外交/涉密/维稳/民族宗教内容
- [ ] 如涉及限制处理内容，是否已添加必要声明和标签
- [ ] 如涉及禁止处理内容，是否已实现自动过滤
- [ ] 敏感关键词过滤机制是否正常工作

### 4. 个人信息保护

- [ ] 变更是否引入了新的个人信息采集或处理
- [ ] 如涉及政策文本中的个人信息，是否已执行脱敏处理
- [ ] 日志输出是否包含个人信息
- [ ] 是否存在个人画像（包括政策相关人物）的逻辑

### 5. 不做预测原则

- [ ] 所有政策研判是否以情景分析形式呈现
- [ ] 不存在确定性预测（如"政策将……"改为"政策可能……"）
- [ ] 涉及机会分析时是否包含风险提示
- [ ] 不存在投资建议或收益保证内容

### 6. 代码质量与测试

- [ ] 代码符合项目 lint 规则（ruff check 通过）
- [ ] 类型检查通过（mypy 检查通过）
- [ ] 相关测试用例已更新或新增
- [ ] 评估测试（eval/）通过且未退化

### 7. 文档完整性

- [ ] 新增功能是否有对应文档
- [ ] API 变更是否更新了接口文档
- [ ] 配置项变更是否更新了配置文档
- [ ] 破坏性变更是否在 CHANGELOG 中记录

### 8. 法律合规

- [ ] 变更是否符合 [法律合规声明](LEGAL_COMPLIANCE.md)
- [ ] 新增数据来源是否符合 [来源准入规则](SOURCE_ADMISSION_RULES.md)
- [ ] 变更是否涉及合规政策的更新需求
- [ ] 跨境数据传输是否已评估合规风险

---

## 评审流程

1. **自动化检查**：CI 自动运行 lint、类型检查、测试
2. **至少 1 位 Reviewer 评审**：按上述 8 项清单逐项检查
3. **敏感内容专项审查**：涉及敏感内容的 PR 须由具有政策分析经验的 Reviewer 复核
4. **合并**：所有检查通过后，由维护者合并

## Reviewer 资格

- 完成至少 3 次被合并的 PR
- 通过项目法律合规培训（自修 [LEGAL_COMPLIANCE.md](LEGAL_COMPLIANCE.md) 并确认）
- 熟悉敏感内容分类与处理规则
