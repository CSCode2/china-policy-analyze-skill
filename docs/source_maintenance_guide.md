# 数据源维护指南

## 1. 概述

本指南说明如何维护中国政策智能系统的数据源注册表，包括新增源、更新源、归档失效源、处理结构变更等工作流程。

## 2. 数据源分级体系

| 级别 | 范围 | 可信度 | 更新频率 |
|------|------|--------|----------|
| S | 中央委员会文件、五年规划、政府工作报告、中央经济工作会议 | 最高 | 实时 |
| A | 国务院文件、各部委、最高法最高检、新华社通稿 | 高 | 每日 |
| B | 省级政府、重点城市政府 | 中 | 每周 |
| C | 市级部门、园区、项目申请信息 | 中低 | 每周 |
| D | 官方解读、发布会实录、专家阐释 | 低 | 按需 |
| E | 商业媒体、自媒体、市场评论 | 最低 | 仅作线索 |
| FP-A | 中国外交部、国防部、国台办等官方外交安全声明 | 高 | 每日 |
| FP-B | 中国贸促会、一带一路网等半官方来源 | 中 | 每周 |
| FP-C | 中国海关、外汇局等执行层面来源 | 中 | 每日 |
| FP-E | 外国政府、国际组织、智库 | 低 | 每周/每月 |

## 3. 新增数据源

### 3.1 准入条件

新增数据源须满足以下所有条件：

1. 必须是公开发布的官方信息渠道
2. 有明确的URL可稳定访问
3. 能提供可解析的HTML或PDF内容
4. 不涉及国家秘密或受控信息
5. 符合 `SOURCE_ADMISSION_RULES.md` 规定的准入标准

### 3.2 添加流程

1. 在 `config/sources.yaml` 中添加新源条目
2. 填写必填字段：name, url, level, topics, crawl_frequency
3. 运行 `scripts/run_smoke_tests.sh` 验证可访问性
4. 创建PR，引用此指南作为变更依据
5. PR合并后，新源在下一个调度周期生效

### 3.3 字段说明

```yaml
- name: 源名称（中文）
  name_en: 英文名称（可选）
  url: 首页或列表页URL
  level: S/A/B/C/D/E/FP-A/FP-B/FP-C/FP-E
  priority: 1-5（1最高）
  topics:
    - 主题标签1
    - 主题标签2
  crawl_frequency: daily/weekly/monthly
  encoding: utf-8（默认）
```

## 4. 更新数据源

### 4.1 URL变更

当数据源URL发生变化时：

1. 更新 `config/sources.yaml` 中的URL
2. 验证新URL可访问
3. 检查页面结构是否变化（需要是否更新解析器）
4. 创建PR并在模板中勾选"Source registry update"

### 4.2 页面结构变更

当源网站改版导致解析失败时：

1. 检查HTML结构变化
2. 更新对应的解析器
3. 运行解析器测试
4. 在PR中说明结构变化和适配方案

### 4.3 内容订阅变更

当数据源新增RSS或Sitemap时：

1. 更新 `config/crawler_rules.yaml`
2. 将入口从列表页爬取改为RSS/Sitemap方式
3. 验证新方式的内容完整性

## 5. 归档失效源

### 5.1 判定失效条件

满足以下任一条件即判定为失效：

- 连续7天HTTP错误（5xx或DNS失败）
- 连续30天无内容更新
- 网站永久关闭或域名过期
- 内容类型不再符合项目需求

### 5.2 归档流程

1. 在 `config/sources.yaml` 中将失效源标记为 `status: archived`
2. 保留历史数据不删除
3. 创建PR并在模板中说明归档原因
4. 如果有替代源，同时添加替代源

## 6. 定期维护任务

### 6.1 每日

- 检查每日抓取错误日志
- 处理新出现的4xx/5xx错误
- 验证关键S级源可访问性

### 6.2 每周

- 审查抓取成功率
- 检查是否有源需要更新URL
- 评估新增源需求

### 6.3 每月

- 全量URL可访问性验证
- 更新源注册表
- 审查源分级是否需要调整
- 清理过期缓存数据
- 校准预测模型

## 7. 紧急处理

### 7.1 高优先级源失效

S级源或FP-A级源失效时：

1. 立即发出告警
2. 2小时内完成排查
3. 24小时内完成修复或找到替代渠道
4. 记录事件到 `cache/incidents/`

### 7.2 内容安全事故

发现抓取到敏感或不当内容时：

1. 立即停止该源的抓取
2. 删除已入库的敏感内容
3. 按SENSITIVE_CONTENT_POLICY处理
4. 记录事件并通知维护人员

## 8. 质量指标

| 指标 | 目标值 | 告警阈值 |
|------|--------|----------|
| 源可访问率 | ≥98% | <95% |
| 日抓取成功率 | ≥95% | <90% |
| 数据新鲜度（90天内） | ≥90% | <80% |
| 解析成功率 | ≥90% | <85% |
| 关键源响应时间 | <5s | >10s |
